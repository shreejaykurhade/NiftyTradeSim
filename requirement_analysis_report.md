# Requirement Analysis Document: Context-Aware Agentic Advisor

## 1. Introduction

### 1.1 Purpose
The purpose of this document is to provide a comprehensive Requirement Analysis and Software Requirements Specification (SRS) for the "Context-Aware Agentic Advisor" project. It defines the system's intended behavior, functional and non-functional requirements, data interfaces, and the core algorithmic paradigms required to execute the multi-agent trading simulation.

### 1.2 Scope
The Context-Aware Agentic Advisor is a simulated trading environment designed specifically for the **Nifty 50 Index**. It bridges the gap between quantitative price action (OHLC) and qualitative informational context (news, macro events). By leveraging a **6-Agent Sequential AI Swarm** (powered by Gemini AI) and a proprietary **5-Dimensional Axis Model**, the system aims to provide hyper-personalized execution advice. The scope is strictly bounded to *simulated* trading interactions for educational and strategy backtesting purposes, meaning no actual financial capital is risked.

### 1.3 Target Audience
This document is intended for software engineers, ML engineers, academic reviewers, and system architects involved in development, deployment, or evaluation of the system.

### 1.4 Evaluation Criteria for Objectives
To ensure the system meets its core objectives, the following evaluation criteria are defined:
*   **Accuracy of 5D State Modeling:** Vectors must accurately reflect the mathematical realities of the underlying market data (e.g., matching standard technical indicators with < 2% margin of error).
*   **Agentic Latency & Uptime:** The RL Monte Carlo predictions and AI Analyst reports must consistently generate within acceptable UX thresholds (< 15 seconds) under normal API load.
*   **Backtesting Efficacy:** The FAISS-based Monte Carlo algorithm must demonstrably retrieve historically relevant KNN (K-Nearest Neighbors) states with over 90% retrieval accuracy.
*   **Hallucination Rate:** The Factual Auditor Agent must reduce LLM hallucination rates to near-zero by cross-referencing generated summaries strictly against retrieved URLs.

## 2. Overall System Description

### 2.1 Product Perspective
The product is an MERN+ stack web application (MongoDB, Express, React.js, Node.js) augmented with real-time WebSocket pipelines and autonomous AI agents. The application operates as a standalone system with external API dependencies (Yahoo Finance, Tavily Deep Search, Google Gemini API).

### 2.2 System Architecture Overview
The system is bifurcated into four core layers:
1.  **Frontend Layer (React 19 SPA):** Handles 5D data visualization using Lightweight Charts and UI interactions.
2.  **Real-time Pipeline (Node.js/Redis/Socket.io):** A high-frequency tick architecture caching live prices in Redis and broadcasting them via Socket.io.
3.  **Agentic Swarm (AI Engine/Gemini):** A sequential pipeline consisting of Scraper, Analyst, Auditor, and Grader roles handling unstructured data ingestion and deductive reasoning.
4.  **Data Persistence (MongoDB):** Long-term storage of user profiles, simulated capital, portfolios, and historical order records.

### 2.3 User Classes and Characteristics
*   **Retail Investor (Primary User):** Interacts with the interface, views dashboard analytics, triggers agentic reasoning for specific tickers, and runs simulated market/limit orders. Their capital constraints and risk profile uniquely shape the advice generated by the system.

### 2.4 Operating Environment
*   **Client:** Any modern web browser (Chrome, Firefox, Safari).
*   **Server Environment:** Node.js backend.
*   **Background Processes:** Persistent cron jobs running every 10 seconds for data polling.

## 3. Functional Requirements

### 3.1 Authentication & Session Management (FR-01)
*   **Description:** The system shall authenticate users securely before providing access to the terminal.
*   **Inputs:** Valid credentials.
*   **Outputs:** Secure JSON Web Token (JWT) establishing an active session.

### 3.2 Real-time Market Data Ingestion (FR-02)
*   **Description:** The system must connect to the `yahoo-finance2` API on a 10-second polling interval to fetch live Nifty 50 ticks.
*   **Processing:** OHLC (Open, High, Low, Close) data must be cached in Redis in-memory storage to maintain <10s average polling latency.
*   **Delivery:** Updates must be broadcasted to all connected clients concurrently via Socket.io (`market_update` event).

### 3.3 Triggering the Sequential Agent Swarm (FR-03)
*   **Description:** The user shall be able to pulse the system to request a deep analysis for a specified Nifty 50 ticker.
*   **Process Initiation:** Transition from 'Live Market Dashboard' to 'Agentic Reasoning Swarm' state logic.

### 3.4 Data Scraping and Information Synthesis (FR-04)
*   **Description:** Upon pulse request, the system must deploy Scraper Agents using the Tavily Deep Search API.
*   **Requirement:** The search must cover a three-tiered context: Domestic News, Sectoral Insights, and Global Macro Trends. A minimum of 15 unstructured sources per stock must be retrieved.
*   **Duplicate Elimination:** To maintain high signal-to-noise ratios, the system must perform an initial deduplication pass. Exact URL matches are immediately discarded. Semantic duplicates (e.g., identical syndicated wire stories across different domains) are eliminated using cosine similarity comparisons on the headline strings before being fed to the LLM context window.

### 3.5 Factual Induction & Deep Analysis (FR-05)
*   **Description:** The system's 'Analyst' Agent must analyze the scraped unstructured data. Rather than looking for keywords, the engine must perform *factual induction*—correlating high-level policies or macro-events to the specific Nifty 50 constituent.

### 3.6 Automated Fact-Checking & Auditing (FR-06)
*   **Description:** An 'Auditor' Agent must cross-reference the generated analysis reporting against the raw scraped data to eliminate AI hallucinations and ensure factual integrity.

### 3.7 Sentiment Grading & 5D Vector Mapping (FR-07)
*   **Description:** The 'Grader' Agent shall translate unstructured reasoning into a structured JSON payload or proprietary score.
*   **Mapping:** The results must automatically be modeled onto a 5D coordinate space: Time (T), Primary Stock (P), Sector (R), Perception (Psi), and Relevance (E).

### 3.8 Hyper-Personalized Execution Advisory (FR-08)
*   **Description:** Final system output must synthesize the 5D sentiment signals directly with the user’s database-persisted Risk Profile and Capital Constraints to output an actionable, context-aware command (Buy/Hold/Short).

### 3.9 Simulated Trade Execution & Portfolio Settlement (FR-09)
*   **Description:** The system shall feature an Order Matching Engine that processes user-initiated Market and Limit orders.
*   **Settlement:** Successful trades must atomically update the user's Portfolio Holdings, Margin constraints, and real-time P&L status within MongoDB and the client view.

## 4. Non-Functional Requirements

### 4.1 Performance & Latency
*   **Market Feeds:** Market ticket latency must aim to be under 10 seconds via the Redis-Socket pipeline.
*   **Agent Execution:** While AI generation inherently faces latency, the synchronous sequential execution of agents should display real-time progress indicators to the user to prevent perceived system unresponsiveness.

### 4.2 Scalability
*   **WebSocket Concurrency:** The Socket.io integration must be capable of broadcasting to multiple retail investor clients concurrently without severe degradation of the polling interval loop.
*   **API Rate Limiting:** The application must gracefully respect rate limits levied by the Yahoo Finance and Tavily external APIs, incorporating retry or failover logic where necessary.

### 4.3 Security & Risk Mitigation
*   **Data Security:** User passwords must be strongly hashed, and API endpoints secured via rate-limiter and authorization middlewares.
*   **Financial Risk:** The system is explicitly designed for simulation. The UI must clearly articulate that no real orders are being piped to a live broker/exchange.

### 4.4 Reliability & Hallucination Mitigation
*   **Institutional Consistency:** The architecture demands the Factual Auditor Check (UC5) state reliably intercepts and flags any analytically generated text that contradicts the scraped source variables before user display.

## 5. Algorithmic and AI Specifications

### 5.1 The 6-Agent Sequential Swarm Dynamics
The core intellectual property of the project resides in breaking down monolithic LLM prompts into specialized cognitive tasks represented by agents:
1.  **Search Orchestrator Agent:** Converts user ticker into optimized Tavily search queries.
2.  **Scraper Swarm:** Specifically targets top-tier financial domain indexing.
3.  **Context Aggregator:** Structures raw outputs into temporal relevance blocks.
4.  **Deep-Analyst Agent:** Performs inductive logic (e.g. mapping "RBI rate hike" to "Automotive Sector contraction").
5.  **Factual Auditor Agent:** The internal safety check; compares Agent 4’s output against Agent 2's raw text.
6.  **Portfolio Grader Agent:** Translates nuanced reasoning into distinct coordinates for the 5D matrix.

### 5.2 The 5D Axis Requirements & Scoring Range
The frontend visualization and backend advisory logic relies strictly on modeling data points accurately across a 5-Dimensional matrix. **Scoring Range:** Every individual dimension is mathematically bounded to a normalized continuous range of `[-1.0, +1.0]`. A score of `-1.0` represents maximum bearishness/contraction, `0.0` is strictly neutral, and `+1.0` represents maximum bullishness/expansion.
*   **Price Action (P):** Normalized daily return versus open.
*   **Volume/Liquidity (V):** Volume surges compared to 14-day moving averages.
*   **Volatility (Vol):** Normalized daily range versus the Average True Range (ATR).
*   **Momentum (M):** Distance and trajectory from Simple Moving Averages (SMA).
*   **Macro/Benchmark (Mac):** Broader market sentiment tracking the Nifty 50 baseline.

### 5.3 Monte Carlo RL & FAISS Backtesting Integration
To compute the final decision from the 5D vector state space, the system utilizes Reinforcement Learning (RL) backed by a FAISS Vector Database.
*   **Historical Data Ingestion:** The system downloads extensive historical OHLCV data (up to 10-20 years of daily candles via Yahoo Finance) to construct a comprehensive historical state database. Each historical day is converted into a 5D/10D state vector and mapped to its forward-looking return (Reward).
*   **FAISS Indexing:** These historical vectors are indexed using high-speed K-Nearest Neighbor (KNN) structures.
*   **Final Decision Computation:** When assessing a live stock, the current 5D state vector is queried against the FAISS database to find the *K* most mathematically similar historical days. A Monte Carlo policy evaluates the historical rewards of those *K* neighbors. If the aggregated probability of the neighbors yields a strong positive expected value (> 0.5 vector sum equivalence), the system outputs a `BUY` signal with a confidence percentage. Conversely, negative expected values yield a `SELL`, and statistically ambiguous neighbor clusters yield a `HOLD`.

## 6. Data Integrity and Interfaces

*   **Yahoo Finance Interface:** Pulls `regularMarketPrice`, `regularMarketVolume`, `regularMarketOpen`, `regularMarketPreviousClose` for Nifty 50 symbols continuously across trading hours via `node-cron`.
*   **Tavily Search Interface:** Uses advanced depth search requiring API authentication; handles requests asynchronously to avoid blocking the main server thread.
*   **Gemini AI Interface:** Expected to receive highly structured JSON prompts and handle responses up to 8k context limits. Must be evaluated using robust retry wrappers to handle occasional 5xx or hallucination errors.
*   **MongoDB Schema:** Must employ transactional or atomic updates for the Wallet/Portfolio collections to prevent race conditions during Order Settlement states.
